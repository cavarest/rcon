= Cavarest RCON Library

image:https://img.shields.io/badge/Java-11%2B-blue[Java Version]
image:https://img.shields.io/badge/Gradle-8.5-green[Gradle Version]
image:https://img.shields.io/badge/License-MIT-yellow[License]

A Java library for communicating with Minecraft servers using the RCON (Remote
Console) protocol. This library provides a simple and efficient API for sending
commands to Minecraft servers and receiving responses.

== Purpose

The Cavarest RCON library allows developers to integrate RCON functionality into
their Java applications for:

* Server administration and automation
* Server monitoring and alerting
* Plugin administration interfaces
* Multi-server management dashboards
* Automated server maintenance scripts

== Features

* Lightweight and efficient implementation
* Full support for RCON protocol (Source RCON)
* Easy-to-use high-level API via `RconClient`
* Low-level API via `Rcon` class for advanced use cases
* Proper handling of RCON packets and responses
* Support for character encoding (UTF-8, ISO-8859-1 for color codes)
* Thread-safe design for concurrent usage
* Comprehensive test suite with unit and integration tests
* Multi-packet response handling with configurable fragment resolution strategies
* Builder pattern for flexible RconClient construction
* Async API using CompletableFuture for non-blocking operations
* Custom exception hierarchy for better error handling

== Architecture

=== Source Code Structure

----
src/main/java/org/cavarest/rcon/
├── Packet.java           # Domain model for RCON protocol packets
├── PacketType.java       # Protocol constants (packet types)
├── PacketCodec.java      # Binary encoding/decoding of packets
├── PacketReader.java     # Low-level packet reading from socket
├── PacketWriter.java     # Low-level packet writing to socket
├── Rcon.java             # Core RCON client with builder pattern
├── RconClient.java       # High-level simplified API
└── exceptions/           # Custom exception hierarchy
    ├── RconException.java              # Base exception
    ├── RconAuthenticationException.java # Auth failures
    ├── RconConnectionException.java    # Connection failures
    └── RconProtocolException.java      # Protocol violations
----

== Usage

=== Quick Start

[source,java]
----
import org.cavarest.rcon.RconClient;
import java.io.IOException;

try (RconClient client = new RconClient("localhost", 25575, "password")) {
    client.connect();
    String response = client.sendCommand("say Hello from RCON!");
    System.out.println("Response: " + response);
} catch (IOException e) {
    System.err.println("RCON error: " + e.getMessage());
}
----

=== Using the Builder Pattern

The `RconClient.Builder` class provides a fluent API for constructing RCON clients:

[source,java]
----
import org.cavarest.rcon.RconClient;
import org.cavarest.rcon.FragmentResolutionStrategy;
import java.io.IOException;
import java.util.concurrent.Executor;
import java.util.concurrent.ForkJoinPool;

RconClient client = RconClient.builder()
    .host("localhost")
    .port(25575)
    .password("mypassword")
    .fragmentResolutionStrategy(FragmentResolutionStrategy.ACTIVE_PROBE)
    .charset(StandardCharsets.UTF_8)
    .asyncExecutor(ForkJoinPool.commonPool())
    .build();

client.connect();
String response = client.sendCommand("list");
----

=== Using the Async API

For non-blocking operations, use the async API methods:

[source,java]
----
import org.cavarest.rcon.RconClient;
import java.util.concurrent.CompletableFuture;

RconClient client = new RconClient("localhost", 25575, "password");
client.connect();

// Async command with default executor
CompletableFuture<String> future = client.sendCommandAsync("seed");
future.thenAccept(response -> System.out.println("Server seed: " + response))
      .exceptionally(e -> {
          System.err.println("Command failed: " + e.getMessage());
          return null;
      });

// Async command with custom executor
Executor executor = Executors.newFixedThreadPool(4);
client.sendCommandAsync("list", executor)
      .thenAccept(response -> System.out.println("Players: " + response));
----

=== Error Handling

The library provides a custom exception hierarchy for better error handling:

[source,java]
----
import org.cavarest.rcon.RconClient;
import org.cavarest.rcon.exceptions.*;
import java.io.IOException;

try {
    RconClient client = new RconClient("localhost", 25575, "password");
    client.connect();
    String response = client.sendCommand("list");
} catch (RconAuthenticationException e) {
    // Authentication failed (wrong password)
    System.err.println("Authentication failed: " + e.getMessage());
} catch (RconConnectionException e) {
    // Connection failed (server down, network issue)
    System.err.println("Connection failed: " + e.getMessage());
} catch (RconProtocolException e) {
    // Protocol violation (invalid response, packet format error)
    System.err.println("Protocol error: " + e.getMessage());
} catch (RconException e) {
    // Generic RCON error
    System.err.println("RCON error: " + e.getMessage());
} catch (IOException e) {
    // Other I/O error
    System.err.println("I/O error: " + e.getMessage());
}
----

=== Using the Low-Level API

[source,java]
----
import org.cavarest.rcon.Rcon;
import java.io.IOException;

try (Rcon rcon = Rcon.connect("localhost", 25575)) {
    if (rcon.authenticate("password")) {
        String response = rcon.sendCommand("list");
        System.out.println("Players: " + response);
    }
} catch (IOException e) {
    System.err.println("RCON error: " + e.getMessage());
}
----

== RCON Protocol Notes

=== Packet Format

All integers are stored in **little-endian** byte order.

|===
| Field name | Field type | Notes

| Length     | int32      | Length of remainder of packet
| Request ID | int32      | Client-generated ID
| Type       | int32      | 3 for login, 2 to run a command, 0 for response
| Payload    | byte[]     | NULL-terminated ASCII text
| 1-byte pad | byte       | NULL

|===

=== Packet Types

* **3 (SERVERDATA_AUTH)**: Authentication request
* **2 (SERVERDATA_AUTH_RESPONSE)**: Authentication response (-1 indicates failure)
* **2 (SERVERDATA_EXECCOMMAND)**: Command execution request
* **0 (SERVERDATA_RESPONSE_VALUE)**: Command response

=== Fragmentation and Multi-Packet Responses

The Minecraft server can fragment responses across multiple packets when the response exceeds the maximum payload size.

Maximum payload sizes are:

* Client to Server: 1446 bytes
* Server to Client: 4096 bytes

==== Automatic Multi-Packet Handling

The library automatically handles multi-packet responses - no configuration required.

When a Minecraft server returns a response exceeding 4096 bytes, the library reads all packets and assembles the complete response transparently. You do not need to configure anything.

[source,java]
----
import org.cavarest.rcon.RconClient;

// Just use the library normally - multi-packet is automatic
try (RconClient client = new RconClient("localhost", 25575, "password")) {
    client.connect();
    String response = client.sendCommand("data get entity @e");  // Handles multi-packet automatically
    System.out.println("Full response received: " + response.length() + " bytes");
}
----

[NOTE]
====
Multi-packet response handling has been verified with responses exceeding 87,000 bytes across ~22 packets.
The library automatically reads all fragments and returns the complete response regardless of size.
====

==== Advanced: Fragment Resolution Strategies

The library uses the `ACTIVE_PROBE` strategy by default, which is reliable across all server implementations. For advanced use cases, you can customize the fragment resolution strategy:

* **ACTIVE_PROBE** (default): Sends an empty command probe to detect the end of the response. Most reliable across all server implementations.
* **TIMEOUT**: Waits for a fixed timeout period after the last packet. Use this for faster detection when server response time is predictable.

[source,java]
----
import org.cavarest.rcon.RconClient;
import org.cavarest.rcon.FragmentResolutionStrategy;

// Advanced: Customize fragment resolution strategy (only if needed)
RconClient client = new RconClient("localhost", 25575, "password");
client.setFragmentResolutionStrategy(FragmentResolutionStrategy.TIMEOUT);
client.connect();
----

=== Character Encoding

Default is UTF-8. Use `StandardCharsets.ISO_8859_1` for legacy servers that use
color codes (section sign `§`).

[source,java]
----
import java.nio.charset.StandardCharsets;

Rcon rcon = Rcon.newBuilder()
    .withChannel(channel)
    .withCharset(StandardCharsets.ISO_8859_1)
    .build();
----

== Testing

This project includes both unit tests and integration tests to ensure code quality and reliability.

=== Unit Tests

Unit tests verify the core functionality of the RCON library without requiring external dependencies:

[source,shell]
----
./gradlew test
----

Unit tests are fast and suitable for local development. Test reports are generated at `build/reports/tests/test/index.html`.

=== Integration Tests

Integration tests verify the library works correctly against a real Minecraft server running in Docker:

[source,shell]
----
./gradlew integrationTest
----

Integration tests require Docker and take longer to run due to:
* Starting a Minecraft Paper 1.21.8 server in a Docker container
* Network initialization and server startup time
* Actual RCON communication with the server

Individual test classes run faster (3-5 seconds each):

[source,shell]
----
# Run specific test class
./gradlew integrationTest --tests "*RconIntegrationTest"

# Run specific test method
./gradlew integrationTest --tests "*LargeResponseTests.shouldHandleGuaranteedMultiPacketResponse"
----

To skip integration tests locally:

[source,shell]
----
SKIP_INTEGRATION_TESTS=true ./gradlew test
----

=== Test Configuration

Tests are organized using JUnit 5 tags:

* *Unit tests*: Tagged with `@Tag("unit")` or no tag, run with `./gradlew test`
* *Integration tests*: Tagged with `@Tag("integration")`, run with `./gradlew integrationTest`

The default `test` task excludes integration tests, ensuring fast local test execution.

=== Creating a Release

Releases are automated using GitHub Actions. To create a release:

==== Option 1: GitHub Release UI (Recommended)

. Go to https://github.com/cavarest/rcon/releases/new[new release page]
. Create a new tag (e.g., `v0.2.0`)
. Add release notes (optional)
. Click "Publish release"
. The `Release` workflow will automatically:
  - Build the project
  - Publish to GitHub Packages
  - Upload JAR to GitHub Releases

==== Option 2: Manual Workflow Trigger

. Navigate to https://github.com/cavarest/rcon/actions/workflows/publish-release.yml[Release workflow]
. Click "Run workflow"
. Enter:
  - `release-version`: e.g., `0.2.0`
  - `post-release-version`: e.g., `0.2.1-SNAPSHOT`
. Click "Run workflow"
. The workflow will:
  - Update version.properties
  - Commit and push changes (creating the tag)
  - Build and publish to GitHub Packages
  - Upload JAR to GitHub Releases

==== Option 3: Push a Git Tag

[source,shell]
----
# Create and push a version tag
git tag v0.2.0
git push origin v0.2.0
----

The `Release` workflow will trigger automatically.
